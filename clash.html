Configuration

In Clash, open the Configuration interface, and you will see an existing basic configuration file named config.yaml. This file serves as Clash‚Äôs startup file, and generally, you should not modify it. If this file contains errors, the Clash core will fail to start. Therefore, we need to import a new user configuration file. These files can be incomplete and usually only need to have the following three fields: proxies, proxy-groups, and rules. There are two methods to import user configuration files: URL download and local file import.

About YAML Files

About YAML Files

YAML stands for ‚ÄúYAML Ain‚Äôt a Markup Language,‚Äù a recursive abbreviation. When creating this language, the original meaning of YAML was ‚ÄúYet Another Markup Language.‚Äù

YAML‚Äôs syntax is similar to other high-level programming languages, and it can elegantly represent data structures like lists, hash tables, and scalar data types. It uses whitespace indentation and relies heavily on appearance, making it particularly suitable for expressing or editing data structures, various configuration files, debugging dumps, and outlining documents (for example, many email header formats resemble YAML closely).

YAML configuration files have the extensions .yml or .yaml.

Basic Syntax:

  • Case-sensitive
  • Uses indentation to represent hierarchical relationships
  • Only spaces, not tabs, are allowed for indentation
  • The number of spaces for indentation doesn‚Äôt matter, as long as elements at the same level are left-aligned
  • ‚Äò#‚Äô denotes a comment

Rule Settings

Open the Proxy page, where you will find four modes: Global, Rule, Direct, and Script. The commonly used mode is Rule, which is equivalent to the familiar PAC mode. The primary purpose of the Proxy page is to switch between proxy modes and nodes.

If you are using the template configuration file mentioned above, you should see a variety of rules in the Rule section. These rules were imported from the template file and can be used directly.

  • Select the node to use in the PROXY section.
  • Final controls the connection mode for URLs not covered by the rules. If server traffic and speed are reasonable, it‚Äôs recommended to choose the PROXY proxy mode to avoid issues with certain websites not loading. To save bandwidth, set it to DIRECT direct connection mode.

Rule Components

Rule Components

A proxy rule mainly consists of three parts:

  • Application target, including full domain (DOMAIN), domain suffix (DOMAIN-SUFFIX), domain keyword (DOMAIN-KEYWORD), IP address/range (IP-CIDR), and GEOIP.
  • The affected IP or domain.
  • The action to take, including direct connection (DIRECT), reject (REJECT), or use a specific proxy group (PROXY, Apple, etc., as defined in the previous steps).

Reference: In-Depth Understanding of Clash Configuration Files

Rule Types

Rule Types

Currently, Clash supports the following rule types:

  • DOMAIN-SUFFIX: Domain suffix matching
  • DOMAIN: Domain matching
  • DOMAIN-KEYWORD: Domain keyword matching
  • IP-CIDR: IP range matching
  • SRC-IP-CIDR: Source IP range matching
  • GEOIP: GEOIP database (country code) matching
  • DST-PORT: Destination port matching
  • SRC-PORT: Source port matching
  • PROCESS-NAME: Source process name matching
  • RULE-SET: Rule Provider rule matching
  • MATCH: Full match

Additionally, the cfw-bypass in the template file controls the IP and domain settings to bypass.

Reference: Rule Editing

Policy Groups

Policy Groups

Policy groups here refer to the proxy-groups in the configuration file. The purpose of policy groups is to distribute incoming requests based on different policies.

Policy groups have the following characteristics:

  • They can contain nodes or other policy groups.
  • They have various types of policies.
  • They serve the rules.

All policy groups serve the rules. If a policy group is not directly or indirectly used in the rules, it becomes unnecessary.

Clash has two built-in policy groups: DIRECT for direct connection and REJECT for blocking.

  • DIRECT: As the name suggests, no proxying is performed, and a direct connection is made. This is suitable for domestic websites and websites not blocked.
  • REJECT: Requests are blocked, suitable for blocking malicious websites, etc.

Additionally, in the template file, we have created policy groups such as Auto Select Fast Nodes and PROXY. You can set corresponding policies in the Proxy interface.

Policy group types include select, url-test, fallback, relay, and load-balance, corresponding to the type in the configuration file.

  • Select: Manual selection, manually choose nodes or policy groups from the node list.
  • Url-test: Latency testing, select nodes with low latency.
  • Fallback: Availability testing, select the first available node in the order of nodes.
  • Load-balance: Load balancing, requests for the same eTLD+1 will use the same proxy.
  • Relay: Relay forwarding, requests sent to this proxy group will be relayed through the specified proxy servers in sequence. Currently, there is no UDP support for this. The specified proxy servers should not contain other relays.

UWP (Universal Windows Platform)

For security reasons, Microsoft employs a sandbox mechanism in UWP applications. By default, they cannot access the loopback address (localhost), which causes applications from the Microsoft Store and other UWP apps to be unable to use the proxy. To address this, the EnableLoopback program is used for modification. This program is already built into Clash for Windows. You can access it by going to the ‚ÄúGeneral‚Äù page, then ‚ÄúUWP App Loopback‚Äù - ‚ÄúStartup Assistant.‚Äù From there, you can select the apps you want to exempt from this limitation and click ‚ÄúSave Changes‚Äù in the upper-right corner.
